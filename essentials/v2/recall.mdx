Recall turns stored context into the *right* context for a specific query. In v2 the entire retrieval surface collapses into a single endpoint  -  [`POST /search`](/api-reference/v2/endpoint/search)  -  that searches [Knowledge](/essentials/v2/knowledge), [Memories](/essentials/v2/memories), or both, with hybrid, text, or vector matching. This page walks through how it works, which knobs matter, and the patterns you'll reach for in production.

For the full request and response schema, see [Search  -  API Reference](/api-reference/v2/endpoint/search-overview).

---

## 1. What is recall

Recall returns the chunks most likely to *help your agent succeed* on a query  -  not just the chunks most textually similar to it. Pure vector search gives you similarity; recall gives you usefulness. Three signals work together to make that happen: semantic similarity, lexical (BM25) matching, and graph traversal across [context graphs](/essentials/v2/context-graphs) HydraDB builds automatically during ingestion.

HydraDB stores two kinds of context in separate stores, and you pick which one (or both) to search via the `source` parameter:

| Store | Contains | Search with |
|---|---|---|
| [**Knowledge**](/essentials/v2/knowledge) | Ingested documents, files, and app sources | `source: "knowledge"` |
| [**Memories**](/essentials/v2/memories) | User preferences, conversation history, per-user inferred content | `source: "memories"` |

A single [`POST /search`](/api-reference/v2/endpoint/search) call can hit either store  -  or both at once via `source: "all"`. When you choose `"all"`, HydraDB runs the two in parallel and returns one merged, re-ranked result set. Set `graph_context: true` and search also pulls in the relevant slice of the context graph alongside the chunks.

## 2. When to use recall

Recall is the right tool when you need:

- **Grounded answers** from documents or app sources.
- **Personalized responses** for a specific user.
- **Context-aware retrieval** across multiple sources.
- **Deterministic filtering** combined with semantic search.

Reach for something else when you need the full document rather than ranked chunks  -  that's what [`GET /source/fetch`](/api-reference/v2/endpoint/fetch-content) is for. If you just want to browse what's been ingested, use [`POST /source/list`](/api-reference/v2/endpoint/list-documents) instead.

## 3. Search dimensions

Every call to [`POST /search`](/api-reference/v2/endpoint/search) has two independent dimensions. Think of them as "what to search" and "how to match."

### `source`  -  what to search

| Value | Searches | Reach for it when |
|---|---|---|
| `"knowledge"` *(default)* | [Knowledge](/essentials/v2/knowledge) | Document Q&A, RAG over shared content |
| `"memory"` | [Memories](/essentials/v2/memories) | Personalization, recalling user preferences |
| `"all"` | Both, merged and re-ranked | Personalized answers grounded in both shared docs and the user's own memories |

### `search_by`  -  how to match

| Value | Pipeline | Reach for it when |
|---|---|---|
| `"hybrid"` *(default)* | Dense vectors + BM25 keyword | General-purpose retrieval and RAG |
| `"text"` | BM25 full-text only | Exact terms, compliance lookups, phrase search |
| `"vector"` | Dense vector only | Pure semantic similarity |

The two dimensions mix freely. `source: "all"` + `search_by: "hybrid"` is the workhorse for personalized RAG; `source: "knowledge"` + `search_by: "text"` + `operator: "phrase"` is the right shape for exact compliance lookups.

---

## 4. How recall works

For `search_by: "hybrid"` and `"vector"`, you also pick a retrieval `mode` that decides how hard HydraDB works to find the best result:

| Mode | Pipeline | Latency |
|---|---|---|
| `"fast"` *(default)* | Single query pass | Lower |
| `"thinking"` | Multi-query expansion + reranking | Higher |

`thinking` expands the query into sub-queries, runs retrieval for each, reranks the combined candidates, and pulls in author-declared related sources (see [`relations` on ingest](/api-reference/v2/endpoint/ingest-content)) into the `additional_context` field. `fast` skips all of that and runs a single pass.

`search_by: "text"` has no `mode` parameter  -  it always runs deterministic BM25 matching.

**Rule of thumb:** if a human will read the result, use `thinking`. If a machine will filter or rerank it downstream  -  or low latency is the priority  -  use `fast`.

## 5. Key parameters

Every parameter on the [`POST /search`](/api-reference/v2/endpoint/search) request, in one place:

| Parameter | Type / values | Purpose |
|---|---|---|
| `tenant_id` <sup style={{ color: "red" }}>*</sup> | string | Tenant identifier  -  the top-level scope. |
| `query` <sup style={{ color: "red" }}>*</sup> | string | Search terms or natural-language question. Cannot be empty. |
| `sub_tenant_id` | string \| null | Scopes recall to a user, workspace, or partition. Defaults to the tenant's default sub-tenant. See [Multi-Tenant](/essentials/v2/multi-tenant). |
| `source` | `"knowledge"` \| `"memories"` \| `"all"` | Which store(s) to search. (default=`"knowledge"`) |
| `search_by` | `"hybrid"` \| `"text"` \| `"vector"` | Retrieval method. (default=`"hybrid"`) |
| `mode` | `"fast"` \| `"thinking"` | Retrieval pipeline. Applies to `hybrid` and `vector` only. (default=`"fast"`) |
| `operator` | `"or"` \| `"and"` \| `"phrase"` | BM25 operator. Applies to `search_by: "text"` only. (default=`"or"`) |
| `max_results` | integer \| null | Maximum chunks to return. Start with 10. |
| `alpha` | float `0.0`–`1.0` \| `"auto"` | Hybrid weight (`1.0` = pure semantic, `0.0` = pure BM25, `"auto"` = HydraDB picks). `hybrid` only. (default=`0.8`) |
| `recency_bias` | float `0.0`–`1.0` | Boost for newer content. (default=`0.0`) |
| `graph_context` | boolean | When `true`, includes the entity/relation graph slice in the response. See [Context Graphs](/essentials/v2/context-graphs). (default=`false`) |
| `metadata_filters` | object \| null | Equality filters applied **before** ranking. Top-level keys match `tenant_metadata` (declared in the [tenant metadata schema](/essentials/v2/metadata)). Nest under `document_metadata` for free-form per-document filters. |
| `additional_context` | string \| null | Extra hint to guide retrieval (e.g. `"User is on the billing page"`). |
| `search_forceful_relations` | boolean | Whether to fetch author-declared related sources into `additional_context`. **Only takes effect in `mode: "thinking"`.** (default=`true`) |

<Warning>
  <strong>Two defaults that surprise developers</strong>
  <ul>
    <li><code>graph_context</code> defaults to <code>false</code>  -  set it to <code>true</code> to receive graph data in the response.</li>
    <li><code>recency_bias</code> defaults to <code>0</code>  -  no recency boost is applied unless you set it.</li>
  </ul>
</Warning>

<Warning>
  <strong><code>metadata_filters</code> needs a declared schema for tenant-level keys</strong>
  <p>Top-level filter keys must be defined in <code>tenant_metadata_schema</code> with <code>enable_match: true</code> at tenant creation. Undeclared filters are silently ignored. To filter on free-form per-document fields, nest them under a <code>document_metadata</code> key. See <a href="/api-reference/v2/endpoint/create-tenant">Create Tenant</a> and <a href="/essentials/v2/metadata">Metadata</a>.</p>
</Warning>

### Tuning heuristics

Most of the time the defaults are right. When they aren't, here's where to start:

- **`alpha`**  -  Start at `0.8`. Lower toward `0.3–0.5` when the query contains literal tokens (error codes, SKUs, product names). Raise toward `0.9` for conceptual questions. Use `"auto"` when query shape varies across calls.
- **`recency_bias`**  -  Leave at `0` for static reference material. Set `0.2–0.4` for mixed content, `0.6–0.8` for changelogs, news, or status updates.
- **`max_results`**  -  Start at `10`. Drop to `5` for tight context windows; raise to `20` if you rerank downstream.
- **`additional_context`**  -  Use it when the query alone is ambiguous. Keep it short and factual.

## 6. Minimal working example

The canonical personalized-answer flow takes a single call: [`POST /search`](/api-reference/v2/endpoint/search) with `source: "all"` returns merged knowledge and per-user memory in one ranked result set.

### Setup

<CodeGroup>

```bash cURL
# Set your key as an environment variable - used in every request below
export HYDRA_DB_API_KEY="your_api_key"
# All requests: -H "Authorization: Bearer $HYDRA_DB_API_KEY" -H "API-Version: 2"
```

```typescript TypeScript SDK
import { HydraDBClient } from "@hydradb/sdk";

const client = new HydraDBClient({
  token: process.env.HYDRA_DB_API_KEY,
});
```

```python Python SDK
import os
from hydra_db import HydraDB

client = HydraDB(token=os.environ["HYDRA_DB_API_KEY"])
```

</CodeGroup>

### One call  -  Knowledge and Memories together

<CodeGroup>

```bash cURL
curl -X POST 'https://api.hydradb.com/search' \
  -H "Authorization: Bearer $HYDRA_DB_API_KEY" \
  -H "API-Version: 2" \
  -H "Content-Type: application/json" \
  -d '{
    "tenant_id": "acme_corp",
    "sub_tenant_id": "user_john_123",
    "query": "How do I reset my password?",
    "source": "all",
    "search_by": "hybrid",
    "mode": "thinking",
    "max_results": 8,
    "graph_context": true
  }'
```

```typescript TypeScript SDK
const result = await client.search.query({
  tenant_id: "acme_corp",
  sub_tenant_id: "user_john_123",
  query: "How do I reset my password?",
  source: "all",
  search_by: "hybrid",
  mode: "thinking",
  max_results: 8,
  graph_context: true,
});
```

```python Python SDK
result = client.search.query(
    tenant_id="acme_corp",
    sub_tenant_id="user_john_123",
    query="How do I reset my password?",
    source="all",
    search_by="hybrid",
    mode="thinking",
    max_results=8,
    graph_context=True,
)
```

</CodeGroup>

### Merge into the LLM prompt

The response is a single `RetrievalResult` containing `chunks[]`, `sources[]`, and  -  when applicable  -  `graph_context` and `additional_context`. Chunks from Knowledge and Memories are already interleaved and ranked by relevance, so no manual merging is required. Pass the result through the helper in [How to Use API Results](/essentials/v2/api-results) to turn it into a context string for your prompt:

```typescript
const context = buildContextString(result);

const completion = await openai.chat.completions.create({
  model: "gpt-4o",
  messages: [
    {
      role: "system",
      content: "Answer using only the provided context. Match the user's preferred style.",
    },
    {
      role: "user",
      content: `${context}\n\nQuestion: How do I reset my password?`,
    },
  ],
});
```

If you need to keep the two streams separate  -  for instance to format user preferences differently from documents in the prompt  -  call [`/search`](/api-reference/v2/endpoint/search) twice in parallel with `source: "knowledge"` and `source: "memories"`, then merge client-side.

### Production checklist

- **Default to `source: "all"`** for personalized answers  -  one call instead of two.
- **Set per-call timeouts.** Generous for `thinking` (3–5s), tight for `fast` (≤500ms).
- **Pass `additional_context`** with known session state (page, feature, role). It sharpens retrieval without extra calls.

## 7. Common mistakes

| Symptom | Cause | Fix |
|---|---|---|
| Empty `query_paths` / `chunk_relations` | `graph_context` not set, or no relations exist for the result set | Set `graph_context: true`. Empty arrays are normal when there's nothing to return  -  see [Context Graphs](/essentials/v2/context-graphs). |
| Recent uploads don't appear in results | Indexing not finished | Poll [`GET /source/status`](/api-reference/v2/endpoint/source-status)  -  chunks are invisible until processing reaches at least `graph_creation`. |
| `metadata_filters` doesn't narrow results | Filter key is undeclared, not match-enabled, or nested wrong | Declare filterable keys in [`tenant_metadata_schema`](/essentials/v2/metadata) with `enable_match: true` before ingestion. For free-form per-document fields, nest under `document_metadata`. |
| Memories missing from a `source: "knowledge"` search | Wrong store selected | Use `source: "memories"` or `source: "all"`. |
| Recency doesn't seem to matter | `recency_bias` defaults to `0` | Set it explicitly between `0.1` and `1.0`. |
| `operator: "phrase"` ignored | `search_by` not set to `"text"` | `operator` only applies to BM25 text search  -  switch `search_by` to `"text"`. |
| `search_forceful_relations` ignored | Request uses `mode: "fast"` | Forceful-relation context is fetched only in `mode: "thinking"`. |

## 8. Advanced patterns

**Hybrid + text in two parallel calls.** When a query mixes a literal token (error code, SKU, function name) with natural-language intent, run `search_by: "hybrid"` and `search_by: "text"` in parallel, dedupe by chunk ID, and treat text hits as a "must include" floor.

**Recall-then-rerank.** Ask for more chunks than you actually need (`max_results: 20`) and apply your own reranker  -  recency windows, compliance filters, business rules  -  before picking the final top-k for the prompt.

**Cache the prompt context.** If the same query repeats inside a session, cache the `RetrievalResult` keyed by `(tenant_id, sub_tenant_id, query, source, search_by)`. Recall is fast, but skipping it entirely is faster.
