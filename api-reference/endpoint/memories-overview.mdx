## Memories vs knowledge

HydraDB distinguishes two classes of ingested content:

| | Memories | Knowledge |
|---|---|---|
| **Endpoint** | `POST /memories/add_memory` | `POST /ingestion/upload_knowledge` |
| **Best for** | User preferences, conversation history, inline notes | Documents, PDFs, app-generated content |
| **Content format** | Text, markdown, conversation pairs | Files, structured app sources |
| **Retrieval** | `POST /recall/recall_preferences` | `POST /recall/full_recall` |
| **Metadata fields** | `metadata`, `additional_metadata` (object) | `metadata`, `additional_metadata` (object) |

Both go through the same processing pipeline (queued â†’ processing â†’ completed) and are searchable via the `boolean_recall` endpoint.

## Key concepts

**Memory** â€“ A single unit of user-specific context. Identified by `source_id`. Created by `POST /memories/add_memory`.

**Inference (`infer`)** â€“ When `true`, HydraDB extracts implicit signals (preferences, entities, relationships) from the content. When `false` (default), content is indexed as-is.

**Conversation pairs** â€“ User-assistant exchanges stored as memory. Useful for capturing chat history with full context.

**Expiry** â€“ Optional TTL in seconds (`expiry_time`). After this duration, the memory is automatically deleted. Useful for ephemeral context.

**Sub-tenants** â€“ Memories are scoped to a sub-tenant. For B2C apps, each end user typically has their own sub-tenant.
